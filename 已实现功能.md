# 已实现功能

本文档用于总结当前这份源码版 AIRI 已经实际落地、可验收、可继续迭代的功能，重点面向你当前这条“定制开发版 AIRI”路线。

说明原则：
- 这里只写已经实现到源码里的能力。
- 不把“未来计划”写成“已经完成”。
- 重点记录与你当前本地体系直接相关的部分。

## 一、源码版基础稳定化

### 1. context bridge 净化修复已落地
- 已在 `packages/stage-ui/src/stores/mods/api/context-bridge.ts` 中加入上下文净化逻辑。
- 已解决一类核心问题：上下文里混入 DOM、Event、Response、File、Blob 等不可安全桥接或不可 `structuredClone` 的对象时，不再直接污染桥接层。
- 当前目标是让桥接层只传递 JSON-safe、可克隆的数据快照。
- 已补对应测试：`packages/stage-ui/src/stores/mods/api/context-bridge.sanitize.test.ts`。

当前效果：
- 减少 “An object could not be cloned” 这类桥接异常。
- 为后续 memory、sidecar、人格层、主动陪伴等补充上下文时提供更稳定的底层基础。

### 2. vision 已改为 App 级常驻运行
- 视觉运行逻辑不再依赖 `devtools/vision.vue` 页面一直存活。
- 已新增共享 runner：`apps/stage-tamagotchi/src/renderer/composables/use-vision-runner.ts`。
- `App.vue` 已接入 vision runner，由更高层负责生命周期。
- 设置页只负责配置。
- devtools vision 页只负责查看、调试、控制，不再是唯一启动源。
- 已加入 `autoRun` 配置项：`packages/stage-ui/src/stores/modules/vision/store.ts`、`packages/stage-pages/src/pages/settings/modules/vision.vue`。

当前效果：
- 不打开设置页或 devtools 页时，vision 也可以后台常驻。
- 关闭设置页后，不会再因为页面卸载而误停 ticker / stream。

## 二、源码版与原本本地数据的衔接

### 3. 源码版用户数据目录已支持优先复用原版 AIRI 数据
- 已新增用户数据路径解析逻辑：`apps/stage-tamagotchi/src/main/app/user-data-path.ts`。
- 在源码预览模式下，会优先尝试使用：
  - `@proj-airi/stage-tamagotchi-source-preview`
  - 若不存在，再回退尝试原版 `ai.moeru.airi`
- 这让源码版更容易复用原本编译版的本地数据目录，而不是每次都像全新 AIRI。

### 4. 已支持从 legacy file origin 恢复本地 localStorage 配置
- 已新增 `apps/stage-tamagotchi/src/renderer/legacy-file-origin-storage.ts`。
- 在源码版运行于 `http://localhost:5173` 且当前配置看起来仍是“空白初始态”时，会尝试从旧的 `file://` origin 读取并恢复关键前端配置。
- 恢复范围包括但不限于：
  - 角色卡
  - 当前激活角色
  - LLM provider 凭据与 provider 启用状态
  - speech 相关部分设置

当前效果：
- 源码版更容易接上你原先在编译版 AIRI 中已配置好的角色卡和模型配置。
- 降低“源码版打开像全新 AIRI”的割裂感。

## 三、桌面端外部集成适配层

### 5. 已建立桌面端独立的“本地协同/外部集成”配置层
- 已新增 desktop-only 的外部集成契约与服务。
- 主体分层已经落地：
  - `apps/stage-tamagotchi/src/shared/external-integrations.ts`
  - `apps/stage-tamagotchi/src/main/services/airi/external-integrations/`
  - `apps/stage-tamagotchi/src/renderer/stores/settings/external-integrations.ts`
  - `apps/stage-tamagotchi/src/renderer/pages/settings/integrations/`
- sidecar / memory 关键配置不再只靠 renderer `localStorage`。
- 已改为桌面端用户数据目录下的独立配置文件持久化。

当前效果：
- 本地协同页面可以稳定保存状态。
- 配置不会因为 dev / preview / file origin 切换而轻易丢失。

### 6. 已支持导入和复用原有 MCP 配置
- 已提供 legacy MCP 导入能力，能够把你原本用户目录中的 `mcp.json` 接入源码版当前配置层。
- 这使得 `filesystem`、`obsidian` 等你原先已配置好的 MCP 服务，可以继续被源码版 AIRI 识别和复用。

当前效果：
- 不需要在源码版里从头重新手配整套本地协同环境。

## 四、外部记忆接入与受控写回

### 7. 已建立 `D:\AIRI-Memory` 的正式接入边界
- 已新增 desktop-only 的 external memory runtime。
- 主要实现位置：
  - `apps/stage-tamagotchi/src/main/services/airi/external-memory/`
  - `apps/stage-tamagotchi/src/shared/external-memory.ts`
  - `packages/stage-ui/src/stores/external-memory-store.ts`
  - `packages/stage-pages/src/pages/settings/memory/index.vue`
- 当前围绕这些固定对象工作：
  - `用户信息.md`
  - `偏好设置.md`
  - `待跟进.md`
  - `近期摘要.md`
  - `角色知识库/`

当前效果：
- AIRI 可以读取外部记忆文件，并在设置页中显示它们的状态和摘要。
- 读取失败时不会直接拖垮主程序，而是降级为该类记忆不可用。

### 8. 主聊天已能读取并使用外部记忆补充上下文
- 已实现 memory supplement 组合逻辑：`packages/stage-ui/src/stores/external-memory.ts`。
- 聊天发送前会尝试刷新并装载 external memory context：`packages/stage-ui/src/stores/chat.ts`。
- 当前会把外部记忆摘要作为 prompt supplement 注入，而不是直接篡改原始 session message 存储结构。

当前效果：
- 顶部状态和设置页中能看到当前读到了哪些记忆片段。
- 主聊天链路已经具备“读取外部记忆并用于回复”的基础能力。

### 9. 已实现受控写回，不再只是只读记忆
- 当前已实现这些写回入口：
  - `writeRecentSummary()`
  - `writeFollowUpItems()`
  - `writeUserProfilePatch()`
  - `writePreferencesPatch()`
- 写回由 `apps/stage-tamagotchi/src/main/services/airi/external-memory/index.ts` 负责实际落盘。
- renderer 只拿 JSON-safe snapshot，不直接操作文件。

当前效果：
- 可以写入近期摘要。
- 可以把新偏好写入 `偏好设置.md`。
- 可以把长期事实写入 `用户信息.md`。
- 可以把待跟进事项写入 `待跟进.md`。

### 10. 已具备基础的自动记忆捕获能力
- 在聊天链路中，用户轮次和助手轮次已经接入 external memory 捕获：
  - 用户发言后会尝试提取偏好、用户事实、待跟进项。
  - 助手回复后会尝试整理近期摘要。
- 触发点位于 `packages/stage-ui/src/stores/chat.ts`。

当前效果：
- 现在不是纯手动写记忆。
- 但目前仍属于“规则提取 + 受控写回”的阶段，还不是高度自治的长期记忆代理。

## 五、纳西妲人格层

### 11. 已建立独立的纳西妲人格补充层
- 已新增共享类型、素材与 store：
  - `packages/stage-ui/src/stores/nahida-persona-shared.ts`
  - `packages/stage-ui/src/stores/nahida-persona.ts`
  - `packages/stage-ui/src/stores/nahida-persona-store.ts`
  - `apps/stage-tamagotchi/src/main/services/airi/nahida-persona/`
- 人格层以当前 `activeCard` 为基线，不直接覆盖原角色卡正文。
- 运行时通过 supplement 追加，而不是改写原角色卡存储。

当前效果：
- 当前角色命中 `纳西妲 / Nahida` 时，可启用人格补充层。
- 不命中时，该层不会套到其他角色上。

### 12. 已支持纳西妲人格三档调节
- 当前已支持：
  - `reserved`
  - `balanced`
  - `active`
- 配置可持久化保存，不依赖单纯的 `localStorage`。
- 设置页可看到：
  - 是否命中纳西妲卡
  - 人格层是否生效
  - 当前摘要说明

当前效果：
- 主聊天已经能在原角色卡基础上，更稳定地呈现“纳西妲味道”。
- 关闭该功能后，可以回到原角色卡基线。

### 13. 纳西妲人格层已接入主聊天与短提醒表达
- 当前人格 supplement 已参与系统提示拼装：
  - `packages/stage-ui/src/stores/character/index.ts`
  - `packages/stage-ui/src/stores/chat.ts`
- `spark notify` 等短反应链路也会复用当前角色系统提示。

当前效果：
- 主聊天和主动提醒的口吻更加一致。
- 人格层主要负责“怎么说”，不负责决定“什么时候来打扰你”。

## 六、主动陪伴协同层

### 14. 已建立桌面端专用的 proactive companion 治理层
- 已新增 desktop-only 的主动陪伴 runtime：
  - `apps/stage-tamagotchi/src/main/services/airi/proactive-companion/`
  - `apps/stage-tamagotchi/src/shared/proactive-companion.ts`
  - `packages/stage-ui/src/stores/proactive-companion-store.ts`
  - `apps/stage-tamagotchi/src/renderer/pages/settings/proactive-companion/`
- sidecar 继续外部运行，AIRI 只做治理与展示，不接管 sidecar 主权。

### 15. 已实现主动事件的筛选、冷却、去重和降级
- 当前 v1 已有这些治理规则：
  - 全局启用开关
  - 全局冷却时间
  - 同主题冷却
  - 来源识别
  - 事件类型归类
  - 放行 / 压制 / 丢弃
- 事件类型已收口到较保守的集合：
  - `gentle-check-in`
  - `reminder`
  - `important`
  - `unknown`

当前效果：
- sidecar 主动事件不再是“只连上但不可见”。
- AIRI 会对其进行克制治理，而不是一股脑全部放行。

### 16. 已支持主动陪伴事件历史与状态查看
- 当前设置页中可查看：
  - sidecar 当前状态
  - 主动陪伴功能是否启用
  - 最近主动事件
  - 每条事件的判定结果
  - 最新呈现方式
  - 最近失败原因或降级原因
- 已支持清空历史、刷新状态等操作。

当前效果：
- 你可以明确看到“最近发生了什么、哪些被放行、哪些被压掉、为什么”。

### 17. 已兼容 legacy `context:update` 主动事件路径
- 除了正式 `spark:notify` 评估路径外，当前还兼容通过 legacy `context:update` 进入的 proactive companion 事件。
- 这类事件现在也能被记录到“最近主动事件”历史中。

当前效果：
- 你已经验证过：触发主动陪伴后，最近事件里能看到对应记录、原因、时间和呈现方式。

## 九、当前已经形成的整体能力

把上面的结果合起来，当前这份源码版 AIRI 已经具备这些实际能力：
- 比原始源码更稳定的 context bridge。
- 不再依赖设置页或 devtools 页常开的 vision 常驻运行。
- 更容易复用原版 AIRI 的本地数据、角色卡和模型配置。
- 可持久化的本地协同配置层。
- 已接入 `D:\\AIRI-Memory` 的外部记忆读取与受控写回。
- 已具备“纳西妲人格补充层”。
- 已具备“主动陪伴治理层”。
- 主动陪伴、记忆、人格三者已经能在 AIRI 主体内形成基础协同。

## 十、当前仍然明确不等于“已经完成”的内容

下面这些方向目前不能算“已经完成”，最多只能算“已经留好入口”或“已有基础”：
- 还没有把 GPT-SoVITS 内置进 AIRI。
- 还没有重构 ComfyUI / 生图链为源码内置体系。
- 还没有把 sidecar 改写为 AIRI 内建系统。
- 还没有把 `D:\\AIRI-Memory` 改造成 AIRI 自己的内置记忆引擎。
- 当前记忆写回虽然已有自动捕获能力，但仍不是高自治长期记忆代理。
- 当前纳西妲人格层是单角色定制增强，不是完整的多角色人格系统。
- 当前主动陪伴是受控协同层，不是高频自治发话系统。

## 十一、适合作为下一步规划的入口

基于当前已实现状态，后续继续规划时，比较自然的入口会是：
- 提升记忆写回判定质量，而不是重新推倒记忆接入层。
- 提升主动陪伴与记忆/人格的协同细腻度，而不是重写 sidecar。
- 继续打磨纳西妲表达质量，而不是立刻扩成多角色平台。
- 在保持外部体系独立的前提下，逐步提升“可日用性”和“低维护成本”。
