# 已实现功能

本文档用于总结当前这份 `airi-nahida` 源码版中，已经真正落地、可验收、可继续扩展的能力。

说明原则：
- 这里只记录已经进源码、已经能跑、已经能解释的内容。
- 不把未来规划写成“已经完成”。
- 优先按“当前可日用价值”而不是按文件清单组织。

## 一、源码稳定化基础

### 1. context bridge 净化修复已落地
- 已在 `packages/stage-ui/src/stores/mods/api/context-bridge.ts` 中加入净化逻辑。
- 当前会过滤或剔除一类不可安全桥接、不可 `structuredClone` 的污染对象。
- 已补对应测试：`packages/stage-ui/src/stores/mods/api/context-bridge.sanitize.test.ts`。

当前效果：
- 减少 `An object could not be cloned` 这类桥接异常。
- 为 memory、persona、sidecar、主动陪伴等后续上下文补充提供更稳定的底层基础。

### 2. vision 已改成 App 级常驻运行
- 视觉运行逻辑不再依赖 `devtools/vision.vue` 页面一直存活。
- 已建立共享 runner：`apps/stage-tamagotchi/src/renderer/composables/use-vision-runner.ts`。
- `App.vue` 负责更高层级挂载。
- settings 页面负责配置。
- devtools vision 页面负责查看、调试、控制。
- 已加入 `autoRun` 配置项。

当前效果：
- 不打开设置页或 devtools 页时，vision 仍可后台常驻。
- 关闭设置页后，不会再因为页面卸载而误停 ticker / stream。

## 二、源码版与原本本地数据的衔接

### 3. 源码版可优先复用原版 AIRI 数据目录
- 已新增用户数据路径解析逻辑：`apps/stage-tamagotchi/src/main/app/user-data-path.ts`。

  - `@proj-airi/stage-tamagotchi-source-preview`

当前效果：
- 源码版不再天然像“全新 AIRI”。
- 更容易继续沿用原本编译版的数据与配置。

### 4. 已支持从 legacy file origin 恢复关键 localStorage 配置
- 已新增 `apps/stage-tamagotchi/src/renderer/legacy-file-origin-storage.ts`。
- 当源码版跑在 `http://localhost:5173` 且当前状态仍像初始空白时，会尝试从旧 `file://` origin 恢复关键配置。

恢复范围包括但不限于：
- 角色卡
- 当前激活角色
- provider 配置与启用状态
- speech 相关部分设置

当前效果：
- 原有角色卡和 LLM 配置更容易迁移回源码版。

## 四、外部记忆层

### 7. 已建立 `D:\AIRI-Memory` 的正式接入边界
- 已新增 desktop-only 的 external memory runtime。
- 主要实现位置：
  - `apps/stage-tamagotchi/src/main/services/airi/external-memory/`
  - `apps/stage-tamagotchi/src/shared/external-memory.ts`
  - `packages/stage-ui/src/stores/external-memory-store.ts`
  - `packages/stage-pages/src/pages/settings/memory/index.vue`

当前围绕这些固定对象工作：

- `偏好设置.md`
- `待跟进.md`
- `近期摘要.md`
- `角色知识库/`

### 8. 主聊天已能读取并使用外部记忆补充上下文
- 已实现 external memory supplement 组合逻辑。
- 聊天发送前会刷新并装载 external memory context。
- 补充内容以 prompt supplement 形式注入，不改原始 session message 存储结构。

当前效果：
- 主聊天已具备“读取外部记忆并用于回复”的基础能力。

### 9. 已实现受控写回，不再只是只读记忆
- 当前已实现这些写回入口：
  - `writeRecentSummary()`
  - `writeFollowUpItems()`
  - `writeUserProfilePatch()`
  - `writePreferencesPatch()`
- renderer 只拿 JSON-safe snapshot，不直接操作文件系统。

当前效果：
- 可以写近期摘要。
- 可以写用户稳定偏好。
- 可以写用户长期事实。
- 可以写待跟进事项。

### 10. 写回结果已增加统一决策字段
- `ExternalMemoryWriteResult` 已增加 `decision`：
  - `written`
  - `skipped-unavailable`
  - `skipped-empty`
  - `skipped-duplicate`
  - `skipped-not-stable`

当前效果：
- 主进程、store、设置页都能清楚知道“这次到底写没写”。
- 记忆写回已经具备基础可解释性。

### 11. external-memory 写回 heuristics 已收紧
- recent summary 改成比较“摘要正文是否实质变化”，不会因为时间头抖动重复写入。
- profile / preferences / follow-ups 已做规范化去重。
- follow-up 删除支持忽略 `[YYYY-MM-DD]` 前缀匹配旧项。
- 读取 `近期摘要` 时会过滤纯标题、纯时间头。
- 读取 `待跟进` 时只把明确动作性 / 提醒性条目放进 prompt 使用内容。

### 12. store 层自动写回已经改为更保守的稳定判定
- `preferences` 只接受明确、稳定的一人称偏好或边界。
- `user-profile` 只接受稳定身份 / 背景事实。
- `follow-ups` 只接受明确提醒 / 后续动作。
- recent summary 仍自动生成，但会先做去重，减少摘要抖动。

当前效果：
- 现在已经不是“什么都写”。
- 更接近“少量高价值写回”。

### 13. settings 页已增加最小可观测增强
- Memory 设置页 recent writes 区块已能显示最近写回的 `decision`。
- 不新增设置入口，不改路由结构。

当前效果：
- 记忆层已经进入“可读、可写、可解释”的状态。

## 六、主动陪伴治理层

### 20. 已建立桌面端专用的 proactive companion 治理层
- 已新增 desktop-only runtime：
  - `apps/stage-tamagotchi/src/main/services/airi/proactive-companion/`
  - `apps/stage-tamagotchi/src/shared/proactive-companion.ts`
  - `packages/stage-ui/src/stores/proactive-companion-store.ts`
  - `apps/stage-tamagotchi/src/renderer/pages/settings/proactive-companion/`
- sidecar 仍保持外部运行，AIRI 只做治理与展示。

### 21. 事件筛选已更严格
- 必须命中配置好的 sidecar source
- sidecar 必须处于 `ready`
- destinations 必须包含 `character` 或 `all`
- headline / note 不能同时为空
- 不满足条件时会被 `suppressed` 或 `dropped`，并记录原因

### 22. 冷却规则已经收口
- 保留 `globalCooldownMs`
- 保留 `topicCooldownMs`
- 保存设置时会做规范化：
  - `globalCooldownMs <= 0` 会回退默认值
  - `topicCooldownMs < globalCooldownMs` 会被抬到至少等于 `globalCooldownMs`

### 23. intensity 已接入治理后的展示强度
- `low`：所有已放行事件都走 `light-prompt`
- `balanced`：只有 `important` 才允许 `prominent-reminder`
- intensity 不改变是否放行，只改变放行后的展示强度

### 24. 历史解释能力已增强
- 每条 `delivered / suppressed / dropped` 都会记录明确 `reason`
- 冷却命中时会记录 `cooldownUntil`
- legacy `context:update` 兼容路径会标成 `legacy:*` 来源
- legacy 记录会明确说明这是兼容记账，不是新的主动触发

### 25. settings 页已升级为治理看板
- 能看到 sidecar 状态
- 能看到最近一次放行时间
- 能看到当前是否还在 cooldown
- 能看到最近 cooldown reason
- 能看到最近 legacy 记录
- 能看到每条历史里的：
  - `rawKind`
  - `urgency`
  - `destinations`
  - `matchedSource`
  - `sidecarReady`
  - `cooldownUntil`
  - `topicKey`

当前效果：
- 这一层已经从“黑箱没反应”提升成“收到事件后能解释为什么没反应”。

### 26. 当前已知限制
- AIRI 这层目前仍然是被动治理层。
- AIRI 只能治理 sidecar 已经发出来的 `spark:notify`。
- AIRI 不能决定 sidecar 什么时候生成事件。
- 当前更大的调试瓶颈是：真实 sidecar 事件样本不足，导致治理层高频实体验证反馈慢。

当前结论：
- proactive companion 治理层本身已经结构化落地。
- 当前剩余问题主要属于“外部事件供给不足”，不是治理层本身黑箱。

  - `persona -> /settings/nahida-persona`

当前效果：

### 31. 第六阶段当前已知限制
- 主进程的 persona target context 仍然需要 renderer 至少先同步一次。
- 之后主进程会缓存并复用该 target context。
- 如果 renderer 长时间不刷新，persona 命中可能仍然基于“最后一次已知值”或空值。
- 这一依赖当前已被明确压缩到 `NahidaPersonaManager.updateTargetContext()` 这一处，不再散落在 coordination manager 内部。

当前结论：
- 第六阶段的结构已经通过验收。
- 这个限制属于“边界清晰、可单独继续优化”的后续项，不再属于本轮阻塞问题。

## 十二、当前仍然明确不等于“已经完成”的内容

下面这些方向目前不能算“已经完成”：
- 还没有把 GPT-SoVITS 内置进 AIRI
- 还没有重构 ComfyUI / 生图链为源码内置体系
- 还没有把 sidecar 改写为 AIRI 内建系统
- 还没有把 `D:\AIRI-Memory` 改造成 AIRI 自己的内建记忆引擎
- 当前记忆写回虽已有自动捕获与稳定判定，但仍不是高自治长期记忆代理
- 当前纳西妲人格层是单角色定制增强，不是完整的多角色人格系统
- 当前主动陪伴是受控协同层，不是高频自治发话系统
