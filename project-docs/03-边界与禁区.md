# 边界与禁区

## 总原则
当前阶段的核心是：**最小修复、稳定迁移、保持兼容、避免破坏**。

任何修改都应优先考虑：
- 是否会破坏当前已经能用的功能
- 是否会扩大范围
- 是否会让后续回退更困难

## 当前明确优先避免的事情
### 1. 不要破坏现有外部系统
当前外部体系已经能在一定程度上工作，因此第一阶段不要轻易破坏：
- GPT-SoVITS
- TTS Adapter
- ComfyUI / 图像服务
- `D:\AIRI-Memory`
- proactive sidecar
- 现有一键启动逻辑

### 2. 不要把第一阶段扩张成“大重构”
第一阶段不是为了：
- 一次性把所有外部服务内置进 AIRI
- 重写 AIRI 框架
- 直接做成完全独立的新智能体系统
- 一口气完成最终形态

### 3. 不要随意污染文件
用户对“文件被污染、乱码、莫名其妙被改坏”非常敏感。
因此应避免：
- 无关文件被一起修改
- 大范围格式化导致 diff 爆炸
- 不必要的编码问题
- 改动过大导致难以回滚

### 4. 不要轻易推翻已验证方向
如果已有补丁方向总体合理，优先：
- 审计
- 修正
- 收尾
- 验证

而不是：
- 直接全部删掉重写
- 在还没验证前换全新方案

## 当前阶段明确不做的事
以下内容可以作为未来候选，但本阶段默认 **不做**：
- 把 GPT-SoVITS 彻底源码内置进 AIRI
- 把 ComfyUI / 生图链彻底内置重构
- 把 proactive sidecar 完整改写进 AIRI 主框架
- 把 Obsidian / 记忆系统做成完全重构
- 把 AIRI 完全解耦成全新独立框架
- 全面替换当前外部启动体系

## 允许做但应谨慎的事
以下事情可以做，但要说明理由并尽量最小化：
- 小范围重构，前提是能明显降低结构问题
- 新增少量辅助文件，用于提高可维护性
- 增加最小必要测试
- 增加设置项，只要不影响现有功能

## 什么时候需要先征求用户
遇到这些情况时，应先暂停并和用户确认：
- 会影响现有外部可运行环境
- 会改启动方式
- 会删除或替换已有功能链路
- 会引入较大架构转向
- 会让后续工作从“收尾修复”变成“重做系统”

## 本阶段判断标准
如果一个改动满足下面任意一条，就要谨慎：
- 不是为了解决当前已知关键问题
- 会显著扩大工作面
- 需要用户承担额外复杂维护成本
- 会增加大量不确定性

一句话：
**先把基础打稳，再谈更大的个性化和深度整合。**
